# references · Leader 人设构建协议

> 本文件是 `skills/create_leader/SKILL.md` 和 `tools/leader_builder.py` 的详细构建规范。
> 定义了如何把「一个真实的人」转化为「可复用的 AI Leader 人格」。

---

## 构建目标

一个好的 Leader 人设，必须能做到：
1. **说话有识别度**——读到第一句话就知道是这个 Leader 说的
2. **反应有逻辑**——在不同场景下，行为是一致的、可预测的
3. **压力有层次**——知道什么情况该轻，什么情况该重
4. **能被用户修正**——当 AI 「演错了」，用户可以说「他不是这样的」并更新

---

## 三件套文件规范

### meta.json 字段规范

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | ✅ | 安全 ID，仅字母/数字/下划线 |
| name | string | ✅ | 显示名（别名） |
| company | string | ✅ | 公司背景 |
| level | string | ✅ | 职级（P9/P10/VP/CTO 等） |
| role | string | ✅ | 职位描述 |
| flavor | string | ✅ | 对应 `flavors/` 中的 flavor ID |
| gender | string | ❌ | male/female/neutral，默认 neutral |
| years_active | string | ❌ | 活跃年份，格式 "2019-2023" |
| tags | array | ✅ | 3-6 个描述性标签 |
| signature_phrases | array | ✅ | 3-8 个标志性句子 |
| mbti | string | ❌ | MBTI 类型，不确定加 ? 后缀 |
| push_style | string | ✅ | 见下方 push_style 枚举 |
| intensity | number | ✅ | 1-5，施压强度 |
| version | string | ✅ | "1.0.0" |
| created_by | string | ✅ | "intake" / "materials_import" / "example" |
| created_at | string | ✅ | "YYYY-MM-DD" |

**push_style 枚举**：
- `直面施压型` — 直接指出问题，不含数量
- `迂回引导型` — 用问题引导，让你自己发现
- `情绪感召型` — 以愿景和使命驱动
- `期望管理型` — 设定高期望，让你不好意思达不到
- `数据驱动型` — 所有判断要有数据支撑
- `流程驱动型` — 强调流程和规范

### persona.md 结构规范

必须包含以下六个部分（顺序不做要求，但都必须有）：

```markdown
## 角色概述      ← 2-3句话，他是谁
## 性格核心      ← MBTI + 3个核心特质
## 标志性行为    ← 3-5个具体行为描述（要具体，不要「很严格」这种抽象词）
## 他的情绪风格  ← 至少4行表格，覆盖好坏情况
## 让他不爽的事  ← 至少3条 bullet
## 他的金句库    ← 至少5句引语，格式 > 「…」
```

**质量标准**：
- 每个标志性行为，读的人能「想象出」那个场景
- 金句库必须是「这个人会说的话」的风格，不是通用标语
- 不好的示例：「他很重视细节」
- 好的示例：「他会在方案 review 后，在 doc 里批注 17 个问题，每个问题都带着行号」

### work.md 结构规范

必须包含以下五个部分：

```markdown
## 会议风格      ← 会前/会中/会后三段，各 1-3 条
## Review 风格   ← 他的 review 流程（3-5步）
## 绩效评价观   ← 至少4级评价标准
## 1-on-1 套路   ← 按对话顺序，5步左右
## 他的「阴暗面」 ← 2-4个真实的缺点 / 操作手段
```

## 从原材料提取的优先级

当有原材料（聊天记录/邮件/描述）时，按以下优先级提取：

```
优先级 1：直接引语（「」里的原话）
  → 直接放入 signature_phrases 和金句库

优先级 2：行为描述（「他会XXX」「他总是XXX」）
  → 提取为 标志性行为

优先级 3：性格评价（「他很XXX」「他是个XXX的人」）
  → 提取为 性格核心 的描述

优先级 4：事件经过（「有一次XXX」「当时XXX」）
  → 提取为 标志性行为 的具体案例

优先级 5：情绪/感受描述（「我当时觉得XXX」）
  → 参考来推断 情绪风格
```

## 修订协议

当用户说「他不是这样的」时，触发修订流程：

1. 找到被否定的字段
2. 询问用户：「那他是怎样的？」或「能给我举个例子吗？」
3. 更新对应字段
4. 输出修订摘要：「我更新了 `{字段}` 为 `{新内容}`」

**不允许「反向覆写」**：用户说「他不是这样的」，不能直接把这个特质删除，要问清楚替代特质是什么。

---

## 版本管理

每次修订，版本号末位 +1（1.0.0 → 1.0.1）。  
主要修订（性格核心改变）中间位 +1（1.0.0 → 1.1.0）。  
完整重建，最高位 +1（1.0.0 → 2.0.0）。
